## 1. 计划目标

本计划不再讨论“应该测什么”的抽象设计，而是把现有联调 e2e 设计拆成一个可以长期执行、持续推进、分阶段验收的实施计划。

本计划必须同时满足以下要求：

1. 每一步都能独立实施、独立审核、独立修复问题。
2. 每一步都有明确的进入条件、审核动作、问题修复闭环和退出条件。
3. 执行过程支持跨周、跨里程碑推进，不依赖一次性完成。
4. 计划本身保持稳定，日常状态更新通过独立台账持续记录。
5. 即使个别分支被阻塞，也不会让整个联调 e2e 工作在中途停下来。

## 2. 执行原则

1. 先收口命令能力，再扩展 e2e 覆盖。凡需要 `disco-cli`、`reg-cli`、`ca-cli` 新能力支撑的场景，必须先把 CLI 能力与文档做对，再落测试。
2. 一步只保护一条主链路。每个步骤只解决一条主要联调链路或一类主要运行时行为，避免把多个高风险面混在一次改造里。
3. 审核独立于实施。每一步的独立审核人不能是该步骤的主要实施人。
4. 问题必须在本步闭环。独立审核发现的问题，原则上必须在当前步骤修复并复审通过，不能带病进入下一步。
5. 证据先于结论。每一步都必须留下测试命令、测试结果、环境前提、审核记录和修复结论。
6. 计划静态、台账动态。计划只在范围变化时调整；日常状态、结论、缺陷、风险一律记录到执行台账。
7. 阻塞分支单独挂账。存在硬依赖的步骤可以进入 `blocked`，但协调人必须立即切换到下一个不受该阻塞影响的 `ready` 步骤，保证推进不断档。

## 3. 过程控制模型

### 3.1 步骤状态机

每个步骤只能处于以下状态之一：

| 状态          | 含义                   | 是否允许进入下一步 |
| ------------- | ---------------------- | ------------------ |
| `planned`     | 已定义范围，但未开始   | 否                 |
| `ready`       | 输入齐备，允许开始实施 | 否                 |
| `in-progress` | 正在实施               | 否                 |
| `in-review`   | 已提交独立审核         | 否                 |
| `fixing`      | 正在修复审核发现的问题 | 否                 |
| `accepted`    | 审核通过，当前步完成   | 是                 |
| `blocked`     | 受外部依赖阻塞         | 否                 |
| `frozen`      | 已冻结，不再修改       | 是                 |

状态流转规则：

1. `planned -> ready`：输入、依赖、负责人和环境前提都已明确。
2. `ready -> in-progress`：实施人开始改代码、文档或测试。
3. `in-progress -> in-review`：实现完成，并附带本步验证证据。
4. `in-review -> fixing`：独立审核发现问题。
5. `fixing -> in-review`：修复完成，提交复审。
6. `in-review -> accepted`：审核通过且无待修复问题。
7. `accepted -> frozen`：相邻依赖步骤开启前，将本步结果冻结。
8. 任一状态可转 `blocked`，但必须记录阻塞原因、解除条件和替代推进动作。

### 3.2 每步必备产物

每一步进入独立审核前，至少要具备：

1. 本步范围说明：本步改什么，不改什么。
2. 变更清单：涉及的仓库、文件、命令、夹具、脚本和文档。
3. 验证证据：测试命令、测试结果、人工检查结论。
4. 审核记录：审核人、审核日期、结论和问题列表。
5. 缺陷记录：问题编号、严重级别、修复状态。
6. 修复结论：所有发现问题已修复、复验通过，或已明确阻塞。

### 3.3 独立审核规则

每一步都必须包含独立审核，标准动作如下：

1. 实施人提交产物后，步骤状态切到 `in-review`。
2. 审核人至少从“契约一致性、行为正确性、测试覆盖、回归风险”四个维度给出结论。
3. 审核发现的问题必须进入缺陷台账，并标注 `critical`、`major`、`minor`。
4. `critical` 与 `major` 问题必须在本步修复后才能验收。
5. `minor` 问题原则上也应在本步修复；若必须后移，需要协调人批准并在台账中显式挂账。
6. 审核通过后，步骤状态切到 `accepted`；冻结前再做一次台账一致性检查。

### 3.4 问题修复闭环

每一步的问题修复必须遵循以下闭环：

1. 发现问题：记录现象、影响范围、复现方式和出现位置。
2. 分类问题：区分实现问题、测试问题、文档问题、环境问题、依赖阻塞问题。
3. 修复问题：只在当前步骤范围内修复，不顺手扩展到下一步范围。
4. 复审问题：由原审核人或指定替代审核人确认问题消除。
5. 关闭问题：记录修复提交、复验结果和关闭时间。

### 3.5 长周期执行与更新设计

为支持长期推进，本计划采用“静态计划 + 动态台账”的双文件模式：

1. 本文件只维护步骤结构、依赖关系和执行规则。
2. `docs/joint-e2e-execution-ledger.md` 负责记录每日推进状态、审核记录、缺陷、风险和决策。
3. 每次步骤状态变化、每次审核结论、每个缺陷状态变化，都必须同步更新台账。
4. 每完成一个 Phase，必须进行一次阶段性回顾，确认下一阶段的 `ready` 队列。
5. 协调人始终维护“1 个进行中步骤 + 至少 2 个 ready 步骤”的滚动队列，防止执行停摆。
6. 当某个高依赖步骤进入 `blocked` 时，协调人必须在同一工作周期内切换到下一个不受阻塞影响的 `ready` 步骤，并在台账中记录切换原因。

## 4. 角色分工

| 角色       | 主要职责                                                 |
| ---------- | -------------------------------------------------------- |
| 实施人     | 完成代码、测试、文档和脚本改造，并提交独立审核           |
| 独立审核人 | 审查契约、实现、测试和回归风险，给出问题清单与验收结论   |
| 协调人     | 维护台账、处理阻塞、安排步骤切换、保证执行持续推进       |
| 负责人     | 做阶段性优先级确认、决定是否冻结 Phase、批准后移或例外项 |

说明：

1. 同一个人可以在不同步骤中承担不同角色。
2. 同一步骤中，独立审核人不能同时担任主要实施人。
3. AI 辅助审查可以参与独立审核，但最终结论仍应由项目指定审核责任人确认。

### 6.6 Phase 5：P2 前置条件跟踪与条件成熟后的高阶场景

| 步骤                               | 实施内容                                                                                | 独立审核                                                                             | 问题修复闭环                                                                 | 退出条件                                |
| ---------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------- | --------------------------------------- |
| 5.1 fanout 前置条件审查            | 审查 `discovery-server` 是否已具备多 peer 调度、分支预算分配和多实例聚合能力            | 审核人检查运行时配置、接口能力和当前实现边界                                         | 若能力仍缺失，则在台账中显式挂为 `blocked` 并登记依赖；不得直接推进 live e2e | 是否可以进入 fanout live e2e 有明确结论 |
| 5.2 fanout live e2e（条件执行）    | 在前置条件满足后，新增多实例 fanout 聚合场景                                            | 审核人检查分支聚合结果、`forwardFanoutLimit`、`forwardFanoutRemaining` 和 route 追踪 | 预算分配错误、结果聚合错误、路由不可追溯的问题必须在本步修复                 | fanout 场景进入高阶联调回归             |
| 5.3 307/508xx 治理场景（条件执行） | 在前置条件满足后，新增 redirect、loop detection、depth/fanout budget 和签名相关错误场景 | 审核人检查协议错误码、错误体和前置构造方式                                           | 错误构造不可靠、状态码或错误体不稳定的问题必须在本步修复                     | ADP 协议治理错误进入回归矩阵            |
| 5.4 DSP 410 fallback（条件执行）   | 在 retention/410 行为稳定后，新增落后窗口触发 snapshot fallback 的场景                  | 审核人检查旧 `last_seq`、`410 Gone` 处理和回到 snapshot 的证据                       | 只测异常不测恢复、snapshot fallback 证据不足的问题必须在本步修复             | DSP 410 fallback 进入高阶联调回归       |

### 6.7 Phase 6：长期维护与持续治理

| 步骤                       | 实施内容                                                             | 独立审核                                                        | 问题修复闭环                                               | 退出条件                                           |
| -------------------------- | -------------------------------------------------------------------- | --------------------------------------------------------------- | ---------------------------------------------------------- | -------------------------------------------------- |
| 6.1 gate / CI 集成         | 根据实际耗时和稳定性，把 P0/P1/P2 分别接入合适的 `just` / CI profile | 审核人检查 gate 分层、执行时间和失败信号是否清晰                | gate 过重、反馈过慢、分层混乱的问题必须在本步修复          | 默认 gate、增强 gate、延迟 gate 分层稳定           |
| 6.2 波动与回归治理         | 建立 flaky case 收敛、失败归档、重跑策略和环境漂移检查               | 审核人检查失败分类、重跑规则和波动台账是否足以支撑长期维护      | 失败重复出现但无归因、环境问题与代码问题混淆必须在本步修复 | 联调 e2e 可以长期运行而不反复失控                  |
| 6.3 定期重基线与下一轮规划 | 每轮阶段性目标完成后，做一次基线回顾、风险清理和下一轮步骤重排       | 审核人检查台账、缺陷、风险、Phase 冻结状态和下一轮 `ready` 队列 | 冻结不彻底、遗留问题未显式挂账、优先级失真必须在本步修复   | 本计划可以长期滚动推进，不需要重新起一个新设计文档 |

## 7. 审核清单模板

每一步的独立审核至少覆盖以下问题：

1. 当前步骤是否仍然只覆盖计划定义的范围，没有扩展到未批准范围。
2. 命令、夹具、文档、脚本、测试和设计约束是否一致。
3. 成功路径、主要失败路径和主要回归路径是否都有证据支撑。
4. 新增场景是否真的通过 CLI 驱动，而不是在测试里重新拼一套 HTTP 细节。
5. 错误输出和失败定位信息是否足够支持下一轮修复。
6. 审核发现的问题是否全部进入缺陷台账并闭环。

## 8. 缺陷分级与处理规则

| 级别       | 含义                                         | 处理要求                                 |
| ---------- | -------------------------------------------- | ---------------------------------------- |
| `critical` | 会破坏主链路、状态机、同步语义或数据正确性   | 当前步骤必须修复，未修复不得验收         |
| `major`    | 会导致契约漂移、测试失真、环境误判或明显回归 | 当前步骤必须修复，复审通过后才能验收     |
| `minor`    | 不影响主链路，但会降低可维护性或可理解性     | 原则上当前步骤修复；若后移必须登记并获批 |

## 9. 执行过程更新设计

### 9.1 日常更新规则

1. 步骤状态变化时，更新执行总表。
2. 每次独立审核结束时，更新审核记录和缺陷台账。
3. 每个缺陷状态变化时，同步更新缺陷台账和最近结论。
4. 新增实现判断、兼容策略或范围调整时，更新决策台账。
5. 新出现的环境风险、跨仓依赖或波动来源，更新风险台账。

### 9.2 节奏设计

1. 每个步骤进入 `in-review` 前，必须完成一次实施人自检。
2. 每个步骤 `accepted` 后，协调人必须在同一工作周期内把下一个步骤推进到 `ready` 或 `in-progress`。
3. 每周至少做一次 Phase 级回顾，检查是否存在长期 `blocked` 项、是否需要调整 `ready` 队列。
4. 每个 Phase 冻结前，负责人必须做一次“范围、缺陷、风险、文档、脚本”一致性确认。

### 9.3 持续推进规则

1. 不允许只维护一个孤立中的 `blocked` 步骤而没有替代动作。
2. 当 P2 类高依赖步骤被阻塞时，协调人应切换到 P0/P1/P4/P6 中仍可推进的维护或稳定化步骤。
3. 每次 Phase 结束后，至少准备下一阶段的两个 `ready` 步骤，避免计划停在阶段边界。
4. 若计划范围需要调整，应先更新本计划，再更新台账；若只是执行状态变化，只更新台账，不反复改计划。

## 10. 当前执行起点

截至 2026-05-03，本计划的起点不是“从零开始”，而是基于以下现状继续推进：

1. `acps-cli` 已有 ontology、CA 被动吊销、discovery 手动 sync 三条有效 e2e 基线。
2. `disco-cli` 已具备结构化 `query` 与 `dsp` 命令组，可作为后续 discovery e2e 的统一前置工具。
3. `docs/disco-cli.md` 已完成命令面同步，适合作为后续步骤的稳定输入。

这意味着执行可以直接从 Phase 0/Phase 1 的收口和 Phase 2/Phase 3 的主链路补齐开始，而不需要重新做一轮纯设计阶段。

## 11. 结论

把联调 e2e 做成长期计划，关键不是一次把所有场景写完，而是建立一套不会中途停下来的推进机制：

1. `reg-cli` 负责 ATR 主链路。
2. `ca-cli` 负责证书生命周期。
3. `disco-cli` 负责 discovery 编排与结构化请求。
4. 每一步都经过独立审核、问题修复和台账更新后再进入下一步。

只要严格按这套计划和台账推进，`acps-cli` 的联调 e2e 就可以持续扩展，而不会在阶段切换或依赖阻塞时停摆。
