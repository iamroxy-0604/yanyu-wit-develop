# acps-cli 统一命令执行计划

状态：Draft

计划日期：2026-05-05

来源：基于 `acps-cli/docs/unified-cli-design.md` 制定，用于指导 `acps-cli` 在单仓内完成统一命令树改造，并支撑长时间自动执行、独立审核和缺陷闭环。

相关文档：

- 设计输入：`acps-cli/docs/unified-cli-design.md`
- 执行台账：`acps-cli/docs/unified-cli-execution-ledger.md`

## 1. 计划目标

本计划不重复解释统一 CLI 的设计目标，而是把设计文档拆解成一个可持续推进、可独立审核、可逐步冻结的实施计划。

本计划必须满足以下要求：

1. 每一步都能独立实施、独立审核、独立修复发现的问题。
2. 每一步都有明确的输入、实施范围、验证动作、审核动作、缺陷闭环和退出条件。
3. 整个过程支持长时间自动执行，不因步骤切换而中断；只有遇到真实阻塞时才允许暂停。
4. 过程控制必须覆盖状态管理、缺陷管理、决策管理、风险管理和阶段冻结。
5. 最终形态直接切换到 `acps-cli` 单入口，不保留旧四个 console script 的兼容壳。

## 2. 范围与边界

本计划的主要修改范围限定在 `acps-cli` 仓内，包括：

1. `pyproject.toml` 中的 console script 与项目元信息。
2. `acps_cli/` 下的 Click 命令树、共享上下文、配置加载和认证材料解析。
3. `tests/` 下的单元、集成、e2e 与帮助信息/命令行为验证。
4. `docs/`、README、联调脚本和示例配置。

本计划默认不主动修改 `registry-server`、`ca-server`、`discovery-server` 的服务端契约；如果实现过程中发现设计无法在现有服务契约上落地，必须把该问题登记为阻塞项，而不是直接扩展本计划范围。

但本计划明确要求在实现后期对整个工作区内所有受版本控制的项目文件执行旧命令清理。清理范围至少包括：`acps-cli`、`registry-server`、`ca-server`、`discovery-server`、`acps-sdk`、`acps-design`、`template-python-project`、`mq-auth-server`、`demo-partner`、`demo-leader`、`acps-infra`、`acps-specs` 中的 README、设计文档、脚本、测试、夹具、示例配置和运行说明。

## 3. 执行原则

1. 先收口契约，再改命令树。凡涉及命令命名、帮助文案、配置键名、认证上下文的修改，必须先与设计文档保持一致，再进入代码实现。
2. 单步收口，不并行放大。当前步骤未通过独立审核前，不开启相邻下游步骤。
   同一 Phase 内的步骤默认串行实施、串行审核，不采用并行推进。
3. 审核独立于实施。每一步都必须由非主要实施人完成独立审核。
4. 问题必须在本步闭环。独立审核发现的问题，原则上必须在当前步骤修复并复审通过，不能带病流转。
5. 证据先于结论。每一步都必须留下实现说明、验证结果、审核记录、缺陷记录和修复结论。
6. 自动推进不中断。某一步 `accepted` 后，必须立即把下一步切到 `ready` 或 `in-progress`，除非遇到登记在案的真实阻塞。
7. 一次性切换。实现过程中不再为旧命令入口、旧配置键或旧帮助路径保留运行时兼容分支。

## 4. 过程控制模型

### 4.1 步骤状态机

每个步骤只能处于以下状态之一：

| 状态          | 含义                         | 是否允许进入下一步 |
| ------------- | ---------------------------- | ------------------ |
| `planned`     | 已定义范围，但尚未开始       | 否                 |
| `ready`       | 输入齐备，允许开始实施       | 否                 |
| `in-progress` | 正在实施                     | 否                 |
| `in-review`   | 已提交独立审核               | 否                 |
| `fixing`      | 正在修复审核发现的问题       | 否                 |
| `accepted`    | 审核通过，当前步完成         | 是                 |
| `blocked`     | 受外部依赖或真实契约缺口阻塞 | 否                 |
| `frozen`      | 已验收并冻结，不再修改       | 是                 |

状态流转规则：

1. `planned -> ready`：本步输入齐备，实施范围与审核人明确。
2. `ready -> in-progress`：实施人开始修改代码或文档。
3. `in-progress -> in-review`：实现完成，并附带本步证据。
4. `in-review -> fixing`：审核发现 `critical` 或 `major` 问题，或要求先修复 `minor` 问题后再复审。
5. `fixing -> in-review`：问题修复完成，重新提交审核。
6. `in-review -> accepted`：审核通过，且不存在未关闭的 `critical` / `major` 问题。
7. `accepted -> frozen`：相邻步骤启动前，把本步产物、审核结论和台账记录冻结。
8. 任一状态均可进入 `blocked`，但必须记录阻塞原因、责任方、解除条件和预计恢复时间。

### 4.2 每步必备产物

每一步进入独立审核前，至少要准备以下产物：

1. 本步范围说明：本步改什么，不改什么。
2. 变更清单：涉及的文件、模块、命令、配置项和测试。
3. 验证证据：命令执行结果、测试结果、人工检查结论。
4. 审核记录：审核人、审核日期、审核意见、结论。
5. 缺陷清单：发现的问题、严重级别、修复状态。
6. 修复结论：所有问题已修复并复审通过，或显式记录阻塞说明。
7. 证据归档位置：
   - 执行总表和审核结论摘要内联写入 `unified-cli-execution-ledger.md`
   - 详细实现/验证/复审证据写入 `docs/unified-cli-execution-evidence/phase-<n>/step-<x>.md`
   - 帮助信息基线使用纯文本 golden 文件，默认存放在 `tests/fixtures/cli-help/`

### 4.3 独立审核规则

每一步都必须包含独立审核，审核动作至少覆盖以下四类维度：

1. 设计契约一致性：命令树、帮助文案、配置键、认证矩阵是否与设计文档一致。
2. 行为正确性：参数解析、错误处理、危险确认、命令路由和状态约束是否正确。
3. 测试与回归：成功路径、失败路径和回归路径是否被覆盖。
4. 范围纪律：是否只修改本步范围，没有把未批准的相邻步骤提前混入。

审核发现的问题必须进入缺陷台账，并按以下级别分类：`critical`、`major`、`minor`。

审核责任链固定为：

1. `GitHub Copilot` 负责实施和修复，不负责给自己的步骤签发独立审核通过结论。
2. `Explore` 子代理负责只读独立审查、问题分级和复审结论，不直接代替实施步骤修复代码。
3. 项目负责人只在 Phase 冻结点和最终整体冻结点进行人工确认，不要求在每一步都介入。

### 4.4 缺陷修复闭环

每一步的缺陷修复必须遵循以下闭环：

1. 发现问题：记录现象、影响范围、发现位置和复现方式。
2. 分类问题：区分设计偏差、实现缺陷、测试缺口、文档漂移、脚本问题。
3. 修复问题：只在当前步骤范围内修复，不顺手扩展新范围。
4. 复审问题：由原审核人或指定替代审核人确认问题已消除。
5. 关闭问题：记录修复提交、复验结果和关闭时间。

替代审核人只能在以下条件下指定：

1. 原审核人在连续 48 小时内不可用，且当前步骤不能因此继续等待。
2. 协调人在决策台账中记录替代原因和替代审核人。
3. 替代审核人必须继续按同一审核清单完成复审。

### 4.5 长时间自动推进规则

为满足“执行过程不要中途停下”的要求，后续自动执行必须遵循下面的推进规则：

1. 当前步骤一旦 `accepted`，必须立即更新台账并启动下一步。
2. 如果审核发现问题，优先留在当前步骤进入 `fixing`，修复后立刻回到 `in-review`，不转向其他步骤。
3. 如果验证失败但问题仍属于当前步骤范围，继续在本步修复，不得提前拆出新 Phase。
4. 只有当问题依赖外部仓契约变更、环境不可用、凭证缺失或工具能力不足时，才允许把步骤置为 `blocked`。
5. 每轮执行至少同步更新一次执行总表、审核记录、缺陷台账、决策台账和下一动作。

自动推进的具体机制固定为：

1. 如果当前执行上下文没有中断，当前步骤 `accepted` 后由同一轮执行直接把下一步切到 `ready` 或 `in-progress`。
2. 如果执行上下文中断，下一轮执行启动时必须先读取台账，并从“最靠前的 `ready` / `in-progress` / `fixing` / `blocked` 步骤”恢复，而不是重新自由选择范围。

### 4.6 范围冻结规则

为避免“相邻步骤并行放大”，每一步从 `in-review` 开始进入范围冻结：

1. 只有本步变更清单中列出的文件允许继续修改。
2. 如果修复审核问题需要新增文件，必须先在审核记录或决策台账中补充说明。
3. 相邻步骤的文件和行为契约不得提前改动，除非当前步骤被显式回退为 `in-progress` 并重新定义范围。

### 4.7 阻塞恢复规则

当步骤进入 `blocked` 后，协调人必须持续维护恢复条件：

1. 每轮新的自动执行开始前，先检查所有 `blocked` 项是否满足解除条件。
2. 满足解除条件后，必须先在台账中记录解除依据，再把状态切回 `ready`。
3. 如果阻塞超过两个执行周期仍未解除，必须把原因和下一步外部依赖写入风险台账。
4. 当实施人和独立审核人都确认“当前设计无法在现有服务端契约上落地”时，实施人可以先把步骤标记为 `blocked` 并登记证据；项目负责人在下一个人工检查点确认是否扩范围或调整设计。

## 5. 角色分工

| 角色       | 主要职责                                       |
| ---------- | ---------------------------------------------- |
| 实施人     | 按步骤完成代码、测试、文档和脚本改造           |
| 独立审核人 | 审核命令契约、行为正确性、测试覆盖和回归风险   |
| 协调人     | 维护状态、处理阻塞、确认范围边界、推动连续执行 |

说明：

1. 同一个人可以在不同步骤里担任不同角色。
2. 同一步骤中，独立审核人不能同时担任主要实施人。

## 7. 分步执行计划

### 7.1 Phase 0：实施准备与基线冻结

| 步骤             | 实施内容                                                                                                   | 独立审核                                                         | 问题修复闭环                                             | 退出条件                            |
| ---------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- | -------------------------------------------------------- | ----------------------------------- |
| 0.1 冻结设计基线 | 复核 `unified-cli-design.md` 中的命令树、认证矩阵、状态机、`run-sync` 边界和配置优先级，确保没有残留未决项 | 审核人逐条对照设计文档，确认无自相矛盾、无关键缺口               | 发现的歧义、冲突或遗漏必须先在文档层修复并复审通过       | 设计基线稳定，可作为唯一实现输入    |
| 0.2 建立执行台账 | 建立执行计划、执行台账、缺陷台账、决策台账、风险台账，并明确角色分工                                       | 审核人检查台账字段、状态定义和更新规则是否足以支撑长时间自动执行 | 字段缺失、状态含义不清、责任分配不明等问题必须在本步修复 | 后续所有 Phase 都可直接在台账中推进 |

#### 7.1.1 Phase 0.1 设计基线冻结检查清单

Phase 0.1 的独立审核必须至少逐项核对以下内容：

1. 根命令树与 `admin` 子树的命名、层级和帮助语义已经定稿。
   对应设计文档：`建议的新命令树`、`命令分配说明`。
   验证标准：逐条核对设计文档中的根命令和 `admin` 子树是否完整、无重名、无别名残留。
2. `agent save` / `submit` / `check` / `sync` / `delete` 的状态机与管理员动作边界已经定稿。
   对应设计文档：`agent` 小节中的状态表与管理员动作说明。
   验证标准：状态表和管理员动作范围没有空白状态、没有互相冲突的允许/禁止组合。
3. `entity derive`、`cert eab fetch`、`cert crl detail`、`admin discovery run-sync` 的命名与定位已经定稿。
   对应设计文档：`entity`、`cert`、`admin discovery` 小节。
   验证标准：命令名、所在命令域和帮助语义在全文中只出现一种说法。
4. `user` / `admin` 两级语义已经定稿，不再引入 `internal` 独立级别。
   对应设计文档：`核心设计原则` 第 4 节、`权限分层建议`、`当前已确认的结论`。
   验证标准：设计文档中不再把 `internal` 当作独立权限级别或独立命令可见性模型。
5. 认证矩阵已经定稿，包括 `admin auth login` 只作用于 Registry 管理员上下文。
   对应设计文档：`认证矩阵`。
   验证标准：每个命令域都有唯一默认身份上下文、唯一默认凭证来源，没有“顺带登录全部 admin 子树”的表述。
6. `base_url` 派生规则已经定稿，包括 `registry.mtls_base_url` 必须显式配置。
   对应设计文档：`配置与凭证设计`。
   验证标准：`registry`、`ca`、`discovery` 的派生规则、例外项和服务根语义在文档中无冲突描述。
7. 配置优先级已经定稿，包括命令行、环境变量、TOML 和派生默认值的顺序。
   对应设计文档：`配置优先级`。
   验证标准：优先级链完整、无重复层级，且不会与认证矩阵冲突。
8. `run-sync` 的禁止行为已经定稿，即不得隐式执行 `dsp start/stop/reset/hard-reset/register-webhook`。
   对应设计文档：`admin discovery` 小节中的职责边界与约束列表。
   验证标准：设计文档已显式写出禁止行为清单，而不是仅依赖实现阶段补充说明。
9. 切换策略已经定稿，即不保留旧四个 console script 和旧配置键运行时兼容逻辑。
   对应设计文档：`落地策略建议`、`当前已确认的结论`。
   验证标准：文档已明确一次性切换，不再包含兼容壳或旧键运行时兼容的计划。

#### 7.1.2 Phase 0 产物与冻结标准

Phase 0 的产物形态和冻结标准固定如下：

1. `0.1` 的主产物是：
   - 已冻结的 `unified-cli-design.md`
   - `docs/unified-cli-execution-evidence/phase-0/step-0.1.md` 中的设计核对记录
   - `unified-cli-execution-ledger.md` 第 3 节中的审核结论摘要

2. `0.2` 的主产物是：
   - 已冻结的执行计划与执行台账
   - `docs/unified-cli-execution-evidence/phase-0/step-0.2.md` 中的台账字段核对记录
   - `unified-cli-execution-ledger.md` 第 8.1 节中的 Phase 0 退出条件状态

3. 只有当 `8.1` 中全部检查项为 `closed` 时，Phase 0 才允许从 `accepted` 进入 `frozen`。
4. 如果 Phase 0 审核发现设计文档仍有关键缺漏，应优先回溯修复设计文档，而不是带着未决项进入 Phase 1。
5. Phase 0 的 `accepted` 判定需要同时满足：
   - `step-0.1.md` 与 `step-0.2.md` 证据文件已生成
   - 审核记录中存在独立审核 `accepted` 结论
   - 台账 `8.1` 的全部检查项为 `closed`

### 7.2 Phase 1：统一入口与共享骨架

| 步骤                     | 实施内容                                                                                                                                                                         | 独立审核                                                               | 问题修复闭环                                                   | 退出条件                                              |

#### 7.5.1 `run-sync` 专项审核清单

Phase 4.3 的独立审核必须显式核对下面这些禁止项和必需项：

1. `run-sync` 的实现只允许读取 `dsp status` / `dsp registry-info`、触发一次 `dsp sync`、轮询 `dsp status`。
2. `run-sync` 不得隐式调用 `dsp start`、`dsp stop`、`dsp reset`、`dsp hard-reset`、`dsp register-webhook`。
3. 当前置条件不满足时，`run-sync` 必须失败退出，并给出转向具体 `dsp` 子命令的提示。
4. `run-sync` 的退出码必须至少区分成功、同步失败、前置条件失败和超时失败。
5. `run-sync` 的帮助文案、实现和测试三者必须表达同一边界，不允许只在文档里限制、代码里放宽。

3. 每次更新 golden 文件，都必须在本步审核记录中说明“为什么变更是预期的”。
4. golden 文件更新属于当前步骤的合法范围内变更，不需要单独开新步骤，但必须经过同一轮独立审核。

1. 当前实现是否与 `unified-cli-design.md` 的命令树、认证矩阵、状态机和配置优先级一致。

4. `acps-cli` 的单元测试、集成测试和端到端测试是否都按当前步骤同步更新；帮助信息是否同步写入 `tests/fixtures/cli-help/` 或等效 golden 基线。

## 9. 缺陷分级与处理规则

| 级别       | 含义                                                 | 处理要求                                 |
| ---------- | ---------------------------------------------------- | ---------------------------------------- |
| `critical` | 会破坏主命令树、认证边界、配置解析或全量回归         | 当前步骤必须修复，未修复不得验收         |
| `major`    | 会导致契约漂移、帮助失真、命令行为错误或明显回归风险 | 当前步骤必须修复，复审通过后才能验收     |
| `minor`    | 不影响主链路，但会降低可维护性或可理解性             | 原则上当前步骤修复；若后移必须登记并获批 |

`minor` 问题如果确需后移，默认只能后移到当前 Phase 的收口步骤（如 `x.4`）或 Phase 5.2 全量质量回归；必须在缺陷台账中保留原编号并标注“延后目标步骤”。

## 10. 台账要求

后续自动执行必须持续维护 `unified-cli-execution-ledger.md`，至少更新以下内容：

1. 执行总表：步骤状态、实施/审核责任、计划/实际时间、最近结论。
2. 审核记录：每轮审核的结论、问题与复审结果。
3. 缺陷台账：问题编号、严重级别、当前状态、修复说明。
4. 决策台账：实现阶段新增的设计判断和范围调整。
5. 风险台账：长期阻塞、回归风险、环境风险。
6. 下一动作：当前应该继续推进的下一步，不允许空置。
