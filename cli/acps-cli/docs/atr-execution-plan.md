# ATR 改造执行计划

状态：Draft

计划日期：2026-05-02

来源：基于 `acps-cli/docs/atr-gap-analysis.md` 制定，用于指导 `registry-server`、`ca-server`、`acps-cli` 三仓协同改造的长期执行过程。

相关文档：

- 执行输入：`acps-cli/docs/atr-gap-analysis.md`
- 设计基线：`acps-design/ATR-DESIGN.md`、`acps-design/ATR-Registry-Server.md`、`acps-design/ATR-CA-Server.md`、`acps-design/ATR-CA-Client.md`

## 1. 计划目标

本计划不重复解释 ATR 目标架构，而是把设计文档拆解成一个可长期执行、可阶段暂停、可逐步验收的实施计划。

本计划必须满足以下要求：

1. 每一步都能独立实施、独立审核、独立修复发现的问题。
2. 每一步都有明确的进入条件、审核动作、缺陷修复闭环和退出条件。
3. 整个过程支持多轮推进，允许跨周或跨迭代执行，不依赖一次性完成。
4. 过程控制必须覆盖状态管理、缺陷管理、风险管理、决策记录和阶段性冻结。

## 2. 执行原则

1. 先收敛契约，再落地实现。凡涉及跨仓 URL、认证、错误语义、配置命名的改动，必须先完成文档基线确认，再进入代码阶段。
2. 单步收口，不并行放大。任一步骤进入“独立审核”前，不开启该步骤的相邻下游步骤。
3. 审核独立于实施。每一步的审核人不能是该步骤的主要实施人。
4. 问题必须在本步闭环。独立审核中发现的问题，原则上必须在当前步骤内修复并复审通过，不能带病流转到下一步。
5. 证据先于结论。每一步都必须留下实现说明、测试结果、审核记录、缺陷清单和修复结论。
6. 文档与实现同步冻结。某一步验收通过时，对应设计文档、实现代码、测试和运维脚本必须保持一致。

## 3. 过程控制模型

### 3.1 步骤状态机

每个步骤只能处于以下状态之一：

| 状态          | 含义                   | 是否允许进入下一步 |
| ------------- | ---------------------- | ------------------ |
| `planned`     | 已定义范围，但未开始   | 否                 |
| `ready`       | 输入齐备，允许开始实施 | 否                 |
| `in-progress` | 正在实施               | 否                 |
| `in-review`   | 已提交独立审核         | 否                 |
| `fixing`      | 正在修复审核发现的问题 | 否                 |
| `accepted`    | 审核通过，当前步完成   | 是                 |
| `blocked`     | 受外部依赖阻塞         | 否                 |
| `frozen`      | 已验收且冻结，不再修改 | 是                 |

状态流转规则：

1. `planned -> ready`：前置输入齐备，范围与负责人明确。
2. `ready -> in-progress`：实施人开始修改代码或文档。
3. `in-progress -> in-review`：实现完成，并附带本步所需证据。
4. `in-review -> fixing`：审核发现问题。
5. `fixing -> in-review`：问题修复后进入复审。
6. `in-review -> accepted`：审核通过，且无待修复问题。
7. `accepted -> frozen`：相邻步骤启动前，将本步结果冻结并记录。
8. 任一状态均可进入 `blocked`，但必须记录阻塞原因、阻塞人、解除条件和预计恢复时间。

### 3.2 每步必备产物

每一步完成实施并提交审核前，至少要准备以下产物：

1. 本步范围说明：本步要改什么，不改什么。
2. 变更清单：涉及的仓库、文件、配置项、接口和测试。
3. 验证证据：测试命令、测试结果、人工检查结论。
4. 审核记录：审核人、审核日期、审核意见、结论。
5. 缺陷清单：发现的问题、严重级别、修复状态。
6. 修复结论：所有发现问题已修复、复验通过，或明确阻塞说明。

### 3.3 独立审核规则

每一步都必须包含一个独立审核过程，标准动作如下：

1. 实施人提交本步产物后，步骤状态切到 `in-review`。
2. 审核人基于“契约一致性、行为正确性、测试覆盖、回归风险”四类维度给出结论。
3. 审核发现的问题必须进入缺陷清单，并标注严重级别：`critical`、`major`、`minor`。
4. `critical` 与 `major` 问题必须在本步修复完成后才能验收。
5. `minor` 问题原则上也应在本步修复；若确有必要后移，必须由审核人与协调人共同批准，并为其创建显式后续步骤或债务项。
6. 审核通过后，步骤状态切到 `accepted`，随后冻结为 `frozen`。

### 3.4 缺陷修复闭环

每一步的缺陷修复必须遵循以下闭环：

1. 发现问题：记录“现象、影响范围、发现位置、复现方式”。
2. 分类问题：区分契约问题、实现问题、测试问题、文档问题、运维脚本问题。
3. 修复问题：只在当前步骤的既定范围内修复，不顺手扩展新范围。
4. 复审问题：由原审核人或指定替代审核人确认问题已消除。
5. 关闭问题：记录修复提交、复验结果和关闭时间。

### 3.5 长周期执行控制

为支持跨周、跨里程碑执行，本计划要求持续维护以下控制项：

1. 执行台账：维护步骤状态、负责人、计划日期、实际日期、阻塞信息。
2. 缺陷台账：维护所有审核发现问题及其修复状态。
3. 决策台账：记录新增设计判断、兼容策略、延后项和例外项。
4. 风险台账：记录跨仓联动风险、环境风险、回归风险和资源风险。
5. 里程碑检查：每完成一个 Phase，必须做一次阶段性回顾与再计划。
6. 冻结点控制：Phase 验收通过后冻结当前基线；下一 Phase 不得直接覆盖上一个 Phase 未结问题。

## 4. 角色分工

| 角色       | 主要职责                                       |
| ---------- | ---------------------------------------------- |
| 实施人     | 按步骤完成设计落地、测试与提交审核             |
| 独立审核人 | 审查契约、实现、测试和回归风险，并提交缺陷清单 |
| 协调人     | 维护状态、处理阻塞、确认范围边界、推动跨仓同步 |

说明：

1. 同一个人可以在不同步骤里担任不同角色。
2. 同一步骤中，独立审核人不能同时担任主要实施人。

## 7. 审核清单模板

每一步的独立审核至少检查以下内容：

1. 设计契约是否与 `atr-gap-analysis.md` 及 `acps-design` 基线一致。
2. 代码改动是否只覆盖当前步骤范围，没有扩散到未批准范围。
3. 配置项、错误码、路径、命令、帮助文本是否一致。
4. 测试是否覆盖成功路径、主要失败路径和回归路径。
5. 文档、脚本、测试和实现是否同步更新。
6. 审核发现的问题是否全部进入缺陷清单并闭环。

## 11. 推荐执行顺序

本计划推荐严格按以下顺序推进：

1. Phase 0：先冻结基线与执行台账。
2. Phase 1：先把 `registry-server` 的双平面和 `/entity` 安全边界做对。
3. Phase 2：再让 `acps-cli` 按设计接入 `9002`，并补齐 `registry-server` 自身 `9002` 服务端证书的本地获取流程。
4. Phase 3：随后补齐 `ca-server` 的内部/管理保护。
5. Phase 4：最后统一收口 URL 命名、账户密钥模型、兼容层和文档脚本。
6. Phase 5：在所有契约和实现稳定后再做全链路验收与发布准备。

这个顺序的原因是：`registry-server` 决定 `9002` 的真实安全边界，`acps-cli` 决定用户态接入方式，`ca-server` 与 `ca-cli` 决定 CA 侧契约收口，三者必须按依赖方向推进，不能倒序。
